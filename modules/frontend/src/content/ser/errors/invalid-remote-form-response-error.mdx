## What this error means and what can trigger it

`ts InvalidRemoteFormResponseError` means the raw remote form response is not one of the two top-level envelopes SER understands. A valid error envelope has `ts type: "error"` plus optional error and status fields. A valid result envelope has `ts type: "result"` plus an optional devalue-encoded `ts data` or `ts result` string.

The raw response may be a primitive, `ts null`, an array, an object without a string `ts type`, an object with an unknown type, or an otherwise incompatible field shape. The class stores that value in `ts envelope`. SER also copies it into the surrounding transport failure's `ts body`.

This differs from `ts UnsupportedRemoteFormResponseError`: the invalid variant means SER could not establish the top-level protocol branch at all.

It is immediately nested inside `ts RemoteTransportError`, so client code normally inspects `ts error.cause` to reach it.

## Why this is the case

SER must choose the error protocol or the result protocol before it can decode any inner value. Reading fields from an unvalidated object would allow malformed or version-skewed responses to drift into later branches and fail with misleading parser errors.

Preserving the raw envelope supports protocol diagnosis while maintaining the classification boundary. No status or domain meaning is invented, because the client has not successfully decoded the structure that would establish either one.
