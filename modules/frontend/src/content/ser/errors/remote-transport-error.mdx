## What this error means and what can trigger it

`ts RemoteTransportError` means the client could not complete or interpret the remote protocol. It is tagged data with optional `cause` and `body` fields. It deliberately has no `status`, because this category includes failures for which SER cannot prove that a usable HTTP response existed.

The tag covers endpoint discovery, malformed form envelopes, missing result slots, payload-schema mismatches, undecodable serialized failures, and generic native promise rejections without a numeric status. Lower-level SER classes such as `ts RemoteFormEndpointMissingError`, `ts InvalidRemoteFormResponseError`, `ts UnsupportedRemoteFormResponseError`, and `ts RemoteErrorDecodeError` normally appear in `cause` rather than directly on the public Effect error channel.

## Why this is the case

Typed remote failures require a successfully decoded error envelope. HTTP failures require a trustworthy numeric status. Transport failures are what remain when the operation breaks before either fact can be established.

Keeping `cause` and `body` separate matters during diagnosis. `cause` records why the protocol failed, often as a specific SER `ts RuntimeError`; `body` records the value that arrived, if anything did. Neither field upgrades the failure into a domain error. The wrapper prevents low-level parser errors, rejected strings, and protocol details from creating an unstable public error union.
