## What this error means and what can trigger it

`ts RemoteHttpError` means a remote operation reached a failure with known HTTP semantics. It is a tagged data record, not a JavaScript `ts Error`. Its required `status` identifies the response class; `body` may preserve a parsed response body, and `cause` may retain the native failure that supplied the status.

SER constructs this variant in three situations. A remote form error envelope may contain a payload that is not one of SER's tagged failures. A native remote helper may reject with an object that exposes a numeric `status`. A non-successful `ts Response` may contain a body that does not decode to an existing tagged failure. In each case an HTTP status is known, even though no more specific typed failure could be recovered.

## Why this is the case

The presence of a status changes what callers can decide. A `ts 401`, `ts 404`, and `ts 503` describe different server outcomes even if their bodies are empty or use an unknown format. Folding those values into `ts RemoteTransportError` would discard information that can drive authentication, absence, or retry behavior.

SER preserves an existing tagged remote failure before using this fallback. The HTTP wrapper is therefore reserved for failures whose status is meaningful but whose body cannot be assigned a more specific SER tag. Only the native normalization path includes the original thrown object as `cause`, because that is the path where such an object exists.
